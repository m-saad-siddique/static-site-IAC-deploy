üöÄ WebGL Deployment Guide

Terraform + S3 + CloudFront with OAC

üìã Overview

This deployment system uses Terraform to provision AWS infrastructure for hosting WebGL applications:

  • S3 Bucket - Private bucket storing your WebGL build files
  • CloudFront Distribution - CDN for fast global delivery
  • Origin Access Control (OAC) - Secure access (S3 bucket is private, only CloudFront can access)
  • IAM Roles/Policies - For automated deployments (optional)

üèóÔ∏è Architecture Flow

Developer - Creates WebGL build
Upload - Files synced to S3 bucket (private)
CloudFront - Serves files via OAC (secure connection)
Users - Access via CloudFront URL (fast, global)
üîí Security: The S3 bucket is completely private. Users can only access files through CloudFront, which uses OAC (Origin Access Control) for secure authentication.

üîß Deployment Methods

Choose the deployment method that fits your workflow:

Method 1

Manual Deployment (AWS Profile)

Best for: Development, testing, small teams

‚úÖ Pros

  • Simple setup
  • No additional configuration
  • Works immediately
  • Good for learning

‚ùå Cons

  • Requires AWS keys
  • Manual process
  • Not suitable for CI/CD

Setup Steps:

  1. Configure AWS profile: aws configure --profile deploy-config
  2. Initialize Terraform: ./scripts/init.sh
  3. Plan deployment: ./scripts/plan.sh dev
  4. Apply infrastructure: ./scripts/apply.sh dev
  5. Upload files: ./scripts/upload.sh dev ./build
# Example workflow aws configure --profile deploy-config ./scripts/init.sh ./scripts/plan.sh dev ./scripts/apply.sh dev ./scripts/upload.sh dev ./WebGLBuild
Method 2

GitHub Actions with OIDC

Best for: Automated CI/CD, production deployments, team collaboration

‚úÖ Pros

  • No AWS keys stored in GitHub
  • Temporary credentials (auto-expire)
  • Fully automated
  • Secure and auditable
  • Scoped to specific repositories

‚ùå Cons

  • Requires initial setup
  • GitHub repository required

How It Works:

GitHub Actions workflow triggers
Requests OIDC token from GitHub
Assumes IAM role in AWS
Gets temporary credentials (1 hour)
Deploys to S3 and invalidates CloudFront

Setup Steps:

  1. Get AWS Account ID: aws sts get-caller-identity --query Account
  2. Edit environments/prod/terraform.tfvars:
    github_actions_oidc = { account_id = "123456789012" repository_filter = "repo:your-username/your-repo:*" }
  3. Apply Terraform: ./scripts/apply.sh prod
  4. Get role ARN: ./scripts/outputs.sh prod | grep deployment_role_arn
  5. Create .github/workflows/deploy.yml:
    name: Deploy on: [push] permissions: id-token: write contents: read jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: aws-actions/configure-aws-credentials@v4 with: role-to-assume: arn:aws:iam::ACCOUNT:role/webgl-prod-deployment-role aws-region: us-east-1 - run: aws s3 sync ./build s3://$(terraform output -raw s3_bucket_id)
üìö See: GITHUB_ACTIONS_SETUP.md for complete workflow examples
Method 3

EC2 Instance with IAM Role

Best for: Long-running deployment servers, automated build systems

‚úÖ Pros

  • No keys needed on EC2
  • Automatic credential rotation
  • Secure (credentials never stored)
  • Good for build servers

‚ùå Cons

  • Requires EC2 instance
  • Ongoing EC2 costs

How It Works:

EC2 instance starts
Automatically gets IAM role credentials
Uses credentials to deploy (no keys needed)
Credentials auto-refresh

Setup Steps:

  1. Edit environments/prod/terraform.tfvars:
    create_deployment_role = true assume_role_services = ["ec2.amazonaws.com"] github_actions_oidc = null # Not needed for EC2
  2. Apply Terraform: ./scripts/apply.sh prod
  3. Launch EC2 instance (via AWS Console or CLI)
  4. Attach IAM role to EC2:
    # In AWS Console: # EC2 ‚Üí Instances ‚Üí Select instance # Actions ‚Üí Security ‚Üí Modify IAM role # Select: webgl-prod-deployment-role
  5. SSH into EC2 and deploy:
    # On EC2 instance (no AWS keys needed!) aws s3 sync ./build s3://bucket-name aws cloudfront create-invalidation --distribution-id DIST_ID --paths "/*"
‚ö†Ô∏è Note: The EC2 instance automatically gets credentials from the attached IAM role. No AWS keys need to be stored on the instance.
